To keep your digital production line running smoothly, you need to know what's going on. Whether that means noticing the big things which may cause systems to grind to a halt, uncovering the minor niggles which can be fixed to prevent problems from escalating, or just for the occasional spot check.

Uhoh comes to your rescue - reaching out to all of the components on your production line and feeding back all of the information you need to maintain quality, trouble-free running.
“Monitoring a large distributed system is a major undertaking, isn't it ?”
Yes, it certainly is. Even with single-digit numbers of hosts running multiple processes, there can be a large number of log files to track, file systems to keep an eye on and processes to monitor. And this is before you've decided how to shape and correlate the raw alarm data collected in order to make sense of it.

One of the main design aims for Uhoh was to produce an ultra-lightweight core so that users can focus on configuring what they would like to monitor without a steep learning-curve to ascend. With all functionality bundled into a single, tiny Java jar file, Uhoh is a doddle to deploy, whether on your own tin, on cloud-hosted virtual machines or embedded in a Docker container. Apart from a Java 1.6+ Runtime, no other software is required.

The complexity of many system monitoring packages on the market is often the reason why organisations fail to deploy effective service and quality assurance tools. The ease of deploying Uhoh can prevent this from happening.
“How is our production line looking today ?”
As Uhoh performs the task of collecting information from your systems, a stream of information is generated. This stream can contain alerts, numerical data (for example performance metrics) or just information messages which can be used to feed other systems with a near real-time flow of quality assurance information.

Each Uhoh server hosts two browser-based user interfaces. The first of which provides a view of currently active alerts stacked by priority (red, amber, green) and time (most recent to the top). The second user interface is used to view charts of metrics plotted over time - for example a historical view of throughput, execution time and error rate for your systems.
“Nuts and bolts”
The key components of an Uhoh monitoring system are:
  • The Client: Client processes run on machines which host components which need monitoring. No configuration files need to reside on Clients as all configuration is requested from the Server.
  • The Server: Server processes collect alert messages from Clients, store all Client configurations and host the browser user interface web server.
“Services - a birds-eye view”
Raw alert and metric data is super-useful in it's own right, but often you really need to know if your organisation's end-to-end services are healthy. With resilient distributed systems there often isn't a straightforward indicator of service availability - it will depend on many factors.

Fortunately, Uhoh again comes to the rescue by virtue of its comprehensive and flexible alert correlation capabilities. Escalating service-impacting issues to the wider organisation, whilst restricting less critical alerts to system support teams is implemented via just a few simple configuration statements.
